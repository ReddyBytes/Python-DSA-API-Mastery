## The Analogy

A hospital pharmacy keeps medications in separate, labeled compartments. The antibiotic shelf and the blood pressure shelf never mix. Each patient gets exactly the dose their doctor prescribed — not whatever happened to be nearby.

Python environments are your compartments. Each project gets its own isolated shelf with exactly the packages it needs, at exactly the versions it was tested with. No contamination between projects. No "it worked before I installed that other thing."

## Why Virtual Environments Exist

Python has one global site-packages directory per installation. Without isolation:

- Project A needs `requests==2.28`
- Project B needs `requests==2.31`
- They cannot both be installed globally at the same time

A virtual environment creates a private copy of the Python interpreter and an isolated `site-packages/` directory for each project. Install whatever you need without affecting anything else on the machine.

```
# Without virtual environment:
/usr/local/lib/python3.11/site-packages/
    requests-2.31/        ← shared by EVERYTHING

# With virtual environment:
project_a/.venv/lib/python3.11/site-packages/
    requests-2.28/        ← only project_a sees this

project_b/.venv/lib/python3.11/site-packages/
    requests-2.31/        ← only project_b sees this
```

## venv — The Built-in Standard

`venv` is included with Python 3.3+. No installation needed.

### Create and activate

```bash
# Create — one-time setup per project
python -m venv .venv                     # ← .venv is the conventional name

# Activate (Mac / Linux)
source .venv/bin/activate

# Activate (Windows)
.venv\Scripts\activate

# Confirm you are in the venv
which python                             # ← should show .venv/bin/python
python --version

# Deactivate
deactivate
```

### Install and track dependencies

```bash
pip install requests flask sqlalchemy    # ← install packages

# Save current state:
pip freeze > requirements.txt           # ← locks ALL installed packages (including transitive)

# Reproduce on another machine:
pip install -r requirements.txt         # ← exact same versions
```

### What venv creates

```
.venv/
├── bin/                 # ← python, pip, activate (Mac/Linux)
│   ├── python -> python3.11
│   ├── pip
│   └── activate
├── lib/
│   └── python3.11/
│       └── site-packages/   # ← your installed packages live here
└── pyvenv.cfg           # ← records the Python version this env was created with
```

Always add `.venv/` to `.gitignore`. Never commit the environment itself — only `requirements.txt` (or `pyproject.toml`).

## Dependency Pinning

### The problem with unpinned requirements

```
# requirements.txt (bad — no pins)
requests
flask
sqlalchemy
```

Run `pip install -r requirements.txt` today: you get flask 3.0.3. Run it six months later: you might get flask 3.1.0 with a breaking change. Your CI passes, production breaks.

### Pinned requirements

```
# requirements.txt (good — exact pins)
flask==3.0.3
requests==2.31.0
sqlalchemy==2.0.30
werkzeug==3.0.3          # ← transitive dep also pinned
click==8.1.7             # ← transitive dep also pinned
```

Every install — local, CI, production — gets identical packages.

### The two-file pattern (best practice)

Maintain two files:

```
# requirements.in  ← you write this: direct deps only, loose versions
flask>=3.0
requests>=2.28
sqlalchemy>=2.0

# requirements.txt  ← generated: all deps, exact pins (never hand-edit)
# Generated by pip-compile
flask==3.0.3
requests==2.31.0
sqlalchemy==2.0.30
werkzeug==3.0.3
```

`requirements.in` is human-authored. `requirements.txt` is machine-generated.

## pip-tools — Automated Pinning

**pip-tools** automates the `requirements.in` → `requirements.txt` workflow.

```bash
pip install pip-tools
```

### Commands

```bash
# Compile: resolve deps and write exact pins
pip-compile requirements.in                  # ← writes requirements.txt

# Upgrade all pins to latest compatible:
pip-compile --upgrade requirements.in

# Upgrade one package:
pip-compile --upgrade-package flask requirements.in

# Install: sync environment to match requirements.txt exactly
pip-sync requirements.txt                   # ← removes packages not in the file too!
```

### Multi-environment files

```bash
# requirements/base.in
flask>=3.0
sqlalchemy>=2.0

# requirements/dev.in
-r base.in                                  # ← includes base
pytest>=8.0
black>=24.0

# requirements/prod.in
-r base.in
gunicorn>=22.0

# Compile all:
pip-compile requirements/base.in -o requirements/base.txt
pip-compile requirements/dev.in -o requirements/dev.txt
pip-compile requirements/prod.in -o requirements/prod.txt
```

## pyenv — Multiple Python Versions

Some projects require Python 3.10. Others need 3.12. Your system might only have 3.11. **pyenv** installs and switches between multiple Python versions on a single machine.

### Install pyenv (Mac)

```bash
brew install pyenv
# Add to ~/.zshrc:
export PYENV_ROOT="$HOME/.pyenv"
export PATH="$PYENV_ROOT/bin:$PATH"
eval "$(pyenv init -)"
```

### Commands

```bash
pyenv install --list                     # ← list all available versions
pyenv install 3.12.3                     # ← install a version
pyenv install 3.10.14

pyenv global 3.12.3                      # ← set default for your user
pyenv local 3.10.14                      # ← pin a version for THIS directory
                                         # ← creates .python-version file
pyenv versions                           # ← list installed versions
pyenv version                            # ← show active version
python --version                         # ← confirm
```

The `.python-version` file:

```bash
# .python-version  ← created by pyenv local
3.10.14
```

Commit `.python-version` to the repo so every developer uses the same Python version.

### pyenv + venv together

```bash
pyenv local 3.12.3                       # ← use Python 3.12 for this project
python -m venv .venv                     # ← creates venv with Python 3.12
source .venv/bin/activate
```

## Poetry — Modern All-in-One

**Poetry** replaces `pip`, `venv`, `pip-tools`, and part of `setuptools` with a single unified tool.

### Install

```bash
pipx install poetry                      # ← recommended: installs in isolation
# or:
curl -sSL https://install.python-poetry.org | python3 -
```

### Create a new project

```bash
poetry new my_project                    # ← scaffolds project structure
cd my_project
```

### Work with an existing project

```bash
poetry install                           # ← install all deps, create .venv
poetry install --without dev             # ← production install (skip dev deps)
```

### Manage dependencies

```bash
poetry add requests                      # ← add to [dependencies]
poetry add "flask>=3.0,<4.0"            # ← with version constraint
poetry add --group dev pytest black     # ← add to [dev] group
poetry remove requests                   # ← remove a dep

poetry update                            # ← update all deps within constraints
poetry update requests                   # ← update one dep
```

### Run commands

```bash
poetry run python src/main.py           # ← run in the managed venv
poetry run pytest                        # ← run tests
poetry shell                             # ← spawn a subshell with venv activated
```

### Build and publish

```bash
poetry build                             # ← creates dist/my_project-0.1.0.tar.gz
                                         #   and dist/my_project-0.1.0-py3-none-any.whl
poetry publish                           # ← upload to PyPI (needs credentials)
```

### pyproject.toml under Poetry

```toml
[tool.poetry]
name = "my_project"
version = "0.1.0"
description = "A production-ready service"
authors = ["Alice <alice@example.com>"]
readme = "README.md"

[tool.poetry.dependencies]
python = "^3.11"
requests = "^2.31"
fastapi = "^0.110"
pydantic-settings = "^2.0"

[tool.poetry.group.dev.dependencies]
pytest = "^8.0"
black = "^24.0"
ruff = "^0.4"
mypy = "^1.10"

[build-system]
requires = ["poetry-core"]
build-backend = "poetry.core.masonry.api"
```

### poetry.lock

Poetry generates `poetry.lock` with exact pins for all packages (including transitive). Always commit `poetry.lock`. Running `poetry install` on any machine reproduces the exact environment.

## conda — Data Science Environments

**conda** is a package and environment manager designed for scientific computing. Unlike pip, it can install non-Python packages (NumPy's C extensions, CUDA libraries, R packages).

### When to use conda over pip

- Machine learning with GPU/CUDA dependencies
- Scientific computing with compiled extensions (NumPy, SciPy, OpenCV)
- Mixed Python + R + C++ projects
- When you need environment-level package management (not just Python packages)

### Commands

```bash
# Create environment
conda create -n my_env python=3.11       # ← named environment
conda create -n ml_env python=3.11 numpy pandas scikit-learn

# Activate / deactivate
conda activate my_env
conda deactivate

# Install packages
conda install numpy                      # ← from conda-forge or defaults channel
conda install -c conda-forge numpy       # ← specify channel

# Export / reproduce
conda env export > environment.yml       # ← export current env
conda env create -f environment.yml      # ← reproduce from file

# List / remove
conda env list                           # ← list all environments
conda env remove -n my_env              # ← delete environment
```

### environment.yml format

```yaml
name: ml_project
channels:
  - conda-forge
  - defaults
dependencies:
  - python=3.11
  - numpy=1.26.4
  - pandas=2.2.1
  - scikit-learn=1.4.2
  - pip:
    - fastapi==0.110.0          # ← pip deps can be embedded too
```

## Tool Comparison Summary

| Feature | venv + pip | pip-tools | poetry | conda |
|---|---|---|---|---|
| Built-in | Yes | No | No | No |
| Env isolation | Yes | Yes | Yes | Yes |
| Dependency resolution | No | Yes | Yes | Yes |
| Lock file | manual | requirements.txt | poetry.lock | environment.yml |
| Packaging/publishing | No | No | Yes | No |
| Non-Python packages | No | No | No | Yes |
| Learning curve | Low | Low | Medium | Medium |
| Best for | Simple apps | Teams on pip | Libraries/apps | Data science/ML |

## requirements.txt vs pyproject.toml

| | requirements.txt | pyproject.toml |
|---|---|---|
| Standard | Older convention | Modern standard (PEP 517/518) |
| Dependency specs | Flat list | Structured with groups |
| Tool config | Separate files | All in one |
| Version ranges | Optional | First-class |
| Used with | pip / pip-tools | Poetry, Hatch, setuptools, uv |
| Lock file | itself (when pinned) | poetry.lock / uv.lock |

For new projects, prefer `pyproject.toml`. For maintaining legacy projects, keep `requirements.txt`.

## pipx — Installing CLI Tools

**pipx** installs command-line Python tools in isolated environments so they do not pollute any project's dependencies.

```bash
pip install pipx

pipx install poetry                      # ← installs poetry in isolation
pipx install black                       # ← black available globally, no conflicts
pipx install cookiecutter

pipx list                                # ← list all installed tools
pipx upgrade poetry                      # ← upgrade one tool
pipx upgrade-all                         # ← upgrade all tools
```

Rule of thumb: install CLI tools with `pipx`, install project dependencies with `pip` or `poetry`.

## Common Mistakes

**Committing the virtual environment:**
`.venv/` contains thousands of files and is machine-specific. Add it to `.gitignore` and share `requirements.txt` or `pyproject.toml` instead.

**Using `pip freeze` for the wrong purpose:**
`pip freeze` captures everything installed, including transitive dependencies and dev tools. For a clean `requirements.txt`, use `pip-compile` from `requirements.in` instead.

**Not pinning in production:**
Unpinned dependencies break silently when a new package version is released. Always pin in `requirements.txt` or use a lock file.

**Mixing global and project installs:**
Running `pip install X` outside a virtual environment installs into your global Python. This causes the exact contamination problem virtual environments solve.
